Building and Monitoring a Credit Score Model

Model Training & Experimentation

Le Thanh Binh  ·  2026

With a clean feature shortlist from EDA and a well-defined target, training begins. This part covers three connected activities: building and iterating on the model (4.1), tracking experiments systematically so results are reproducible and comparable (4.2), and auditing the model rigorously before it is declared production-ready (4.3). In a regulated industry like banking, the audit step is not optional — it is the basis on which model governance will approve or reject deployment.

4.1

Training

Start with a logistic regression scorecard

The traditional credit scorecard is a logistic regression model applied to WOE-transformed features. Despite being decades old, it remains the standard in many banks — not because it is the most accurate, but because it is interpretable, auditable, and accepted by regulators. Each feature's contribution to the final score is a simple additive points system: a borrower with a monthly income between 10M–20M VND gets +25 points; a bureau score below 500 gets −40 points. The total score maps directly to an approval band.

Start here, even if the eventual model will be a gradient-boosted tree. The logistic regression gives you a meaningful baseline, validates that the WOE-transformed features carry real signal, and produces a model that the risk team can reason about without a black-box explanation tool.

Time-based train/validation/test split

  • Dataset is split strictly by origination date, not randomly — ensures forward-time generalization
  • Typical structure for 3-year dataset: months 1–24 (train), months 25–30 (validation), months 31–36 (test)
  • Test set is held out and touched only once, at final evaluation — prevents information leakage
  • Time-based structure reflects production reality: model scores applications submitted today based on patterns in the past
Fig — Time-based split. TRAIN learns, VAL selects the model, TEST is touched exactly once. Dashed border = held-out. No random shuffle permitted.

Feature pipeline inside a transformer

  • All preprocessing — WOE encoding, missing value imputation, outlier clipping, scaling — implemented in scikit-learn Pipeline
  • Pipeline is the serialized artifact deployed to production, not just model weights — ensures identical train/serve transformations
  • Any transformation outside the pipeline becomes a potential source of training-serving skew
  • Validates that the feature definitions are identical between development and production environments

Logistic Regression vs. Gradient Boosting

Logistic Regression
  • Interpretable: each feature's WOE-based points are additive
  • Auditable: regulators accept the methodology and output
  • Baselined: 5–10 Gini points lower than GBDT, but predictable
  • Low governance cost: minimal explanation documentation required
LightGBM / XGBoost
  • Non-linear: captures feature interactions automatically
  • Higher accuracy: typically 5–10 Gini points above logistic regression
  • Requires SHAP: explanation tools needed for governance review
  • Governance cost: additional audit burden, documentation complexity

In banking, model complexity is a governance cost, not just a technical one. A gradient boosting model that improves Gini by 3 points over a logistic regression may not be worth the additional audit burden if the bank's risk committee requires manual review of the explanation method. Calibrate complexity to the institutional context.

4.2

Experiment Tracking

Credit scoring projects often run for months, with multiple data scientists running parallel experiments on different feature sets, model families, and hyperparameter configurations. Without systematic experiment tracking, the team quickly loses the ability to reproduce results or understand why a particular model version performed better than another.

What to log for every experiment

  • Dataset version: date range, row count, and target distribution used for training
  • Features: complete feature list and preprocessing configuration (WOE bins, imputation strategy)
  • Hyperparameters: all tuning parameters (learning rate, tree depth, regularization coefficients)
  • Metrics: AUC, Gini, KS, and log loss on train / validation / test sets
  • Model artifact: serialized model path and framework version (scikit-learn, LightGBM)
  • Metadata: training timestamp, environment variables, and experimenter name — enables reproduction

Model versioning

  • Register each candidate model in a model registry with semantic versioning: v1.0, v2.0, v2.1
  • Tag models by status: development, staging, production, retired
  • When a new version is promoted to staging, the previous staging version is automatically demoted
  • Prevents ambiguity: the model registry is the single source of truth for which version is active
4.3

Model Audit

Standard validation metrics (AUC, Gini, KS on the test set) are necessary but not sufficient before deploying a credit model. A structured audit covers three additional dimensions: adversarial robustness, stability analysis, and subgroup performance.

✓
Adversarial Testing
Edge cases: missing bureau data, zero income, extreme outliers. Define expected behavior before production.
✓
PSI Analysis
Population Stability Index measures score distribution shift. Flag if PSI > 0.1 or > 0.2 pre-deployment.
✓
KS / Gini
Ensures discrimination power. Target: KS > 30%, Gini > 0.40 across all segments.
✓
Subgroup Performance
Evaluate metrics by product, customer segment, geography. Flag segments below threshold.
✓
Leakage Re-check
Verify no future information in features. Re-audit feature definitions and temporal boundaries.
✓
Governance Report
Document all findings — favorable and unfavorable — with remediation actions or accepted risk.

Adversarial testing

Construct synthetic applicant profiles that test the model at its edges. Example test cases:

  • Missing bureau data: What score does the model assign when bureau features are completely null? Does it degrade gracefully or produce extreme predictions?
  • Zero income: Input error scenario. Verify the model assigns a conservative score or explicitly rejects the application
  • Extreme income: Applicants with declared income 10x the training maximum. Ensure prediction is reasonable, not extrapolated
  • Age outliers: Applicants below 18 or above 100 years. Model should handle boundary cases without errors
  • Zero credit age: Brand new applicants with no credit history. Verify fallback logic (default conservative score, manual review flag)

Edge cases are not hypothetical — they will appear in production, and the model's behavior in these cases should be defined and tested before deployment, not discovered by a loan officer at 10 PM on a Friday.

PSI — Population Stability Index

PSI measures how much the distribution of a variable has shifted between two populations. In model audit, it compares the score distribution in the training set against a recent out-of-time validation window:

PSI = Σ (Actual% − Expected%) × ln(Actual% / Expected%)

PSI < 0.1
No meaningful shift. Training distribution still represents current population. Model remains valid.
0.1 ≤ PSI ≤ 0.2
Moderate shift. Investigate cause. May be temporary market conditions or early warning of model drift.
PSI > 0.2
Significant shift. Training data no longer representative. Refresh dataset and retrain before deployment.

KS Statistic

The KS statistic is the maximum difference between the cumulative distribution functions of good and bad borrowers across the score range. It measures the model's ability to separate the two classes at its most discriminating point. For retail credit scorecards, a KS above 30% is generally considered acceptable. The KS chart also reveals where in the score range the model discriminates best — which informs where to set the approval cutoff for maximum effectiveness.

Fig 1 — KS chart showing cumulative score distributions of good vs. bad borrowers. KS = max vertical separation. Target: KS > 30%.

Subgroup performance

  • Evaluate model metrics across key segments: loan products (personal loan, credit card, cash advance)
  • Customer segments: new-to-bank vs. existing, salaried vs. self-employed
  • Geographic regions if applicable — some models show regional bias
  • A portfolio Gini of 0.50 may mask a Gini of 0.30 on self-employed segment, below acceptable threshold — resolve before deployment
Fig 2 — Subgroup Gini across customer segments. Self-employed segment (red) falls below minimum threshold of 0.45 — requires remediation before deployment.

The model audit report is the primary deliverable for model governance review. It should document all findings — including unfavorable ones — with the remediation actions taken or the rationale for accepting the finding as a residual risk. Governance committees review the integrity of the process, not just the headline metrics.
