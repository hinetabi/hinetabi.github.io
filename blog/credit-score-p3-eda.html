Building and Monitoring a Credit Score Model

EDA: Data Source Quality

Le Thanh Binh  ·  2026

After the data source register is built in Part 2, EDA is where the data tells you whether your assumptions were correct. In credit scoring, this step has a specific character: you are not just exploring the data to understand its shape — you are running a systematic quality gate that determines whether the data is safe to train on. Skipping or rushing this stage is one of the most reliable ways to build a model that looks good in development and fails in production.

Data Quality

Quality Checks on the Credit Dataset

Missing values

In credit data, missingness is rarely random. A null bureau score usually means the applicant is new-to-credit — which is a meaningful signal in itself, not just a gap to fill. A missing declared income field could indicate a self-employed applicant who was not required to submit payslips, or it could indicate a data pipeline failure. The first case should be imputed thoughtfully (e.g., median income of the same product segment); the second should trigger a data quality escalation.

For each column, compute the null rate by time period and by product type. A sudden spike in nulls in a particular month often traces back to a system migration, a field being made optional in the loan application form, or an upstream ETL job that broke. These are not model problems — they are data engineering problems that must be resolved before training.

Class imbalance

Default rates in retail banking are typically between 2% and 10% depending on the product and market. A 95:5 good-to-bad ratio is common. This matters because naive training on an imbalanced dataset produces a model that predicts "good" for almost everything and achieves 95% accuracy while being essentially useless for its actual purpose. Standard approaches include class weighting (passing class_weight='balanced' in scikit-learn), oversampling the minority class with SMOTE, or adjusting the decision threshold post-training. For a credit scorecard, class weighting is usually preferred because it preserves the true score distribution without introducing synthetic samples that could distort the score calibration.

Schema validation

Credit data flowing from multiple systems is prone to silent type errors: a numeric income field arriving as a string, a date stored as a Unix timestamp in one system and as a formatted string in another, a categorical field with inconsistent encodings across vintages (e.g., employment type encoded as "FT" in 2021 and "Full-time" in 2022). These are caught by comparing the schema of recent data against the schema documented at the source audit stage. Any drift here must be resolved at the ETL layer before it reaches model training.

Outliers

Income and loan amount fields in credit applications frequently contain extreme outliers — either genuine high-net-worth applicants or data entry errors (someone entering their annual income as their monthly income, or a system that stored values in a different currency before a migration). Winsorization at the 1st and 99th percentile is a common default, but the right approach depends on the feature. For bureau balance fields, a genuinely high balance is important signal and should not be clipped arbitrarily. Always inspect the top and bottom 1% of each numeric field manually before deciding on outlier treatment.

Feature Analysis

Feature-Level Checks

Target leakage

Target leakage is the most dangerous EDA finding in credit scoring, and it is more common than it should be. Leakage occurs when a feature used at training time incorporates information that would not be available at the time of the actual credit decision. Classic examples in banking include: using a feature that is computed from data generated after the loan was disbursed (e.g., number of missed payments in the first 3 months, included as a "feature" by mistake), or including the loan amount itself when the loan amount was adjusted based on a prior risk assessment that incorporated the default probability.

A practical heuristic: any feature that has an abnormally high Information Value (IV > 0.5) on the first pass should be scrutinized for leakage. Legitimate features rarely achieve IV that high. If one does, understand exactly what it measures and when it is generated — before it makes it into the feature set.

Temporal consistency

Credit data has a natural time axis: applications come in over months and years, and the "bad" label is determined with a lag (DPD90+ within 12 months). This means the dataset must be split by time. Training on data from 2020–2022 and validating on 2023 data is the correct setup. Shuffling randomly — which is the default in most ML libraries — leaks future data into the training set and produces optimistically biased validation metrics. The mistake is easy to make and hard to detect after the fact.

Always validate the temporal integrity of the dataset by plotting your default rate by origination month. If default rates appear artificially low for recent vintages (the last 12 months), it is likely because those loans have not had enough time to go bad yet — the performance window is incomplete. These recent vintages should be excluded from training or treated as an indeterminate population.

Distribution analysis

Plot the distribution of each numeric feature, segmented by target class (good vs. bad). Income distributions for bad borrowers tend to be left-shifted relative to good borrowers — but the overlap region is where the model earns its value. A feature where the two distributions are nearly identical carries no discriminatory power. Weight of Evidence (WOE) binning is the standard credit risk approach for this: it transforms a continuous feature into log-odds bins, making the monotonic relationship between the feature and default probability explicit and interpretable.

Correlation and redundancy

In a credit dataset with dozens of bureau attributes, it is common to find groups of highly correlated features (e.g., number of open credit cards, total revolving balance, credit utilization ratio — all three capture aspects of the same underlying behavior). High inter-feature correlation does not hurt tree-based models much, but it can cause instability in logistic regression-based scorecards. Use variance inflation factor (VIF) checks during feature selection and prefer retaining the feature with the highest IV within a correlated cluster rather than all of them.

Output

EDA Report and Feature Shortlist

The output of EDA is a feature shortlist and an EDA report documenting the quality findings, decisions made, and risks flagged. The feature shortlist should include: the final set of candidate features, the null rate and imputation strategy for each, the WOE/IV analysis summary, any features excluded and why (leakage, low IV, compliance), and any data quality issues that were resolved or that remain open as accepted risks.

This document is not just for the data scientist — it is the evidentiary record that the risk team and model governance committee will review before approving the model for production. In a regulated environment, the transparency of this stage is not optional.
