Building and Monitoring a Credit Score Model

EDA: Data Source Quality

Le Thanh Binh  ·  2026

EDA in credit scoring is not pure exploration — it is a systematic quality gate. After the data source register is built in Part 2, this step determines whether each source is safe to train on. Missing here is one of the most reliable paths to a model that passes dev review and quietly fails in production.

EDA output is also regulatory evidence. The risk team and model governance committee will review this stage before approving the model. Transparency is not optional.

Data Quality

Quality Checks on the Credit Dataset

Missing values

Missingness in credit data is rarely random — each null often carries its own meaning and its own fix.

Intentional nulls
  • Null bureau score → new-to-credit applicant
  • Missing payslip income → self-employed segment
  • Treat as a signal, impute with segment median
Pipeline failures
  • Sudden null spike in a specific month
  • Likely cause: ETL breakage, field made optional
  • Escalate as data engineering issue before training
How to investigate
  • Compute null rate by time period and product type
  • Plot null rate trend — spikes indicate incidents
  • Cross-reference with system migration logs
Resolution
  • Document every null decision in the EDA report
  • Unresolved nulls = accepted risk, must be noted
  • Do not drop rows without understanding why

Class imbalance

Default rates in retail banking sit between 2–10%. A naive model that predicts "good" for every applicant scores 93% accuracy but zero utility. Three standard remedies:

  • Class weighting — class_weight='balanced' in scikit-learn. Preferred for scorecards because it preserves the true score distribution.
  • SMOTE oversampling — generates synthetic minority samples. Useful for tree models but can distort score calibration.
  • Threshold calibration — keep training as-is, adjust the decision cutoff post-training to match business risk appetite.
Fig 2 — Class imbalance in a typical retail credit portfolio. 93% accuracy means nothing when the model predicts "good" for everyone.

Schema validation

Credit data from multiple systems is prone to silent type drift. Key checks to run against the source audit schema:

  • Numeric fields arriving as strings — income, loan amount, bureau balance
  • Inconsistent date formats — Unix timestamp vs. formatted string across vintages
  • Categorical encoding drift — e.g. "FT" in 2021 becoming "Full-time" in 2022
  • Undocumented new columns — new fields added upstream without notifying the model team

Schema drift caught here must be resolved at the ETL layer before training begins — not patched inside the model pipeline.

Outliers

Income and loan amount fields frequently contain extreme values — either genuine high-net-worth applicants or data entry errors. The right treatment depends on the field type:

  • Application fields (declared income) — winsorize at 1st / 99th percentile by default
  • Bureau balance fields — a genuinely high balance is predictive signal, don't clip blindly
  • Always inspect the top and bottom 1% of each numeric field before deciding on treatment
  • Common data errors: income stored as monthly value when system expects annual; currency unit mismatch after migration
Feature Analysis

Feature-Level Checks

Target leakage

Leakage is the most dangerous EDA finding in credit scoring — and more common than it should be. A feature leaks if it incorporates information that would not exist at the time of the actual credit decision.

Any feature with an Information Value > 0.5 on first pass should be treated as a leakage suspect. Legitimate features rarely reach this level. Audit the exact generation timestamp of that field before letting it into the feature set.

  • Classic example 1: "Number of missed payments in first 3 months" included as a feature — it's generated after the loan disbursed
  • Classic example 2: Loan amount adjusted based on a prior risk score, then used as input — a circular dependency
  • Detection: for every feature, document the event timestamp of its generation and compare to the loan application timestamp
  • Leaky model symptom: near-perfect metrics during training, dramatic decay on real-world holdout

Temporal consistency

Credit data has a natural time axis. The "bad" label is determined with a 12-month lag (DPD90+ within 12 months of disbursement). This forces a specific train/validation structure.

✓ Correct: time-based split
  • Train: 2020 – 2022 applications
  • Validation: 2023 applications
  • Future never leaks into past
  • Reflects real production conditions
✗ Wrong: random shuffle
  • Train/val split by random seed
  • 2023 loans appear in training set
  • Validation metrics are optimistically biased
  • Discovered only after production degradation
Fig 3 — Correct time-based split. The dashed cutoff line ensures no 2023+ data appears in the training set.

Recent vintages (last 12 months) show artificially low default rates — the performance window is incomplete. Exclude them from training or treat them as an indeterminate population.

WOE / IV Analysis

Weight of Evidence (WOE) binning transforms a continuous feature into log-odds bins, making the relationship between that feature and default probability explicit. Information Value (IV) summarises the overall predictive power of a feature.

Fig 4 — Interactive WOE analysis. Bars above the zero line = good borrower signal; bars below = bad borrower signal. Toggle between features to compare predictive power.

IV interpretation guide:

  • IV < 0.02 — Useless (do not include)
  • 0.02 – 0.1 — Weak predictor
  • 0.1 – 0.3 — Medium predictor
  • 0.3 – 0.5 — Strong predictor
  • IV > 0.5 — Suspiciously strong — audit for leakage

Correlation and redundancy

Credit bureau data often yields clusters of correlated features. Three typical clusters in retail credit:

  • Utilization cluster — open credit cards, total revolving balance, credit utilization ratio (all measure the same behaviour)
  • Delinquency cluster — DPD 30+, DPD 60+, DPD 90+ counts over various time windows
  • Income cluster — declared income, bank transaction income estimate, employer-implied salary band

For logistic regression scorecards, check Variance Inflation Factor (VIF) within each cluster. Keep the single feature with the highest IV per cluster; drop the rest to avoid instability in the model coefficients.

Output

EDA Report and Feature Shortlist

The EDA stage produces two deliverables: a feature shortlist and a written EDA report that the risk team and model governance committee will review.

  • 1 Null-rate log — null rate by feature, time period, and product segment; imputation decision for each
  • 2 Schema diff report — deviations from the source audit schema, resolved vs. unresolved
  • 3 WOE / IV table — all candidate features ranked by IV, with WOE bin charts for the top 20
  • 4 Leakage audit log — features investigated for leakage, evidence reviewed, outcome (keep / exclude)
  • 5 Feature shortlist — final candidate set with justification; features excluded and why (leakage, low IV, compliance)
  • 6 Vintage chart — default rate by origination month, flagging vintages excluded due to incomplete performance window
