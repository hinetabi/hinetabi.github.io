Missing Values — When Absence Is the Signal

Le Thanh Binh · Data Scientist, Cake by VPBank · April 2026

EDACredit RiskData Quality

Most EDA guides treat missing values as a nuisance to be cleaned up — fill with the mean, move on. In banking, that approach can be more dangerous than leaving the values missing. The reason a field is empty is often as informative as any value it could contain.

Section 01What Are Missing Values & Why Do They Matter?

What is it?

Missing values are fields that exist in the schema but have no recorded value for a given row. They appear as NaN, None, empty strings, or — more insidiously — as disguised sentinels like -999, 0, or 9999 that a system writes when it has no real value to report.

Why does it matter in banking?

Consider three examples from our loan dataset, all technically "missing" but each requiring a completely different response:

Missing credit_score: The bureau returned no score. This customer has no credit history — a thin-file borrower. Filling with the population mean score treats them as average-risk, when their actual risk profile is meaningfully different. The missing value is a segment label, not a gap to fill.

Missing collateral_type: The customer offered no collateral. This is not unknown — it is a definitive "None." If the field is imputed with the most common collateral type, you are inventing secured lending where none existed, corrupting one of the strongest default predictors in the dataset.

Missing annual_income: Could be a data entry failure, a self-employed customer with irregular income who cannot document a fixed amount, or a deliberate omission. The same missing value has three completely different implications requiring different actions. Business context is required before any decision.

Why "fill with mean" is dangerous in banking: SBV Circular 11 and Basel model validation requirements demand that missing data treatment be documented and justified. "We filled with the mean" is not acceptable if that decision introduces bias against a protected group — for example, if thin-file customers (disproportionately younger or lower-income) are systematically assigned average scores that overstate their creditworthiness, leading to over-lending at rates that cause harm.

Section 02Three Types of Missingness

Before deciding what to do with missing data, you must understand why it is missing. The three types are fundamentally different and require different treatments.

MCAR — Missing Completely At Random
The probability of a value being missing has no relationship to any other variable — not to the missing value itself, not to any other feature.
Example: A system outage on one day caused interest_rate to not be recorded for loans disbursed on that day. The affected loans are not systematically different from non-affected ones. Simple mean imputation is acceptable here — but MCAR is rare in banking data.
MAR — Missing At Random (Conditional)
The probability of a value being missing is related to other observed variables, but not to the missing value itself. Given those other variables, the missingness is random.
Example: Self-employed customers are more likely to have missing annual_income (because income is irregular and harder to document) — but among self-employed customers, the income values that are recorded are representative of what is missing. Imputing using employment-type-specific medians is appropriate here.
MNAR — Missing Not At Random
The probability of a value being missing is directly related to the value that is missing. The absence encodes information that cannot be recovered by imputation.
Example: High-risk customers are less likely to have a credit_score (because they have never borrowed before, or because previous lenders did not report to the bureau). The missing score itself correlates with higher default risk. You cannot impute this — you must flag it as its own segment: has_credit_history = False.

Section 03Missing Rates in Our Loan Dataset

Visualizing missing rates across all features at once reveals two things: how much data is absent, and whether the missingness pattern has a structure. Features that are missing together — for example, credit_score and num_late_payments both missing for the same customers — suggest a shared upstream cause (no bureau record exists for these customers at all).

Missing Value Rates — All Features
Green < 5% (clean) · Amber 5–30% (investigate) · Red > 30% (critical decision needed)

Pattern insight: credit_score (45% missing) and num_late_payments (12% missing) are co-missing — the same customers tend to have both absent. This confirms a shared cause: no bureau record exists for these borrowers. Any imputation strategy applied to one should consider its relationship to the other.

Section 05Decision Flowchart

How to Decide What to Do With Missing Data
Follow from top to bottom for each missing feature

The documentation requirement: Every missing value decision must be recorded in the Model Development Document with a rationale. For each affected feature: state the missing rate, the missingness type (MCAR/MAR/MNAR), the decision made, and the business justification. This is not optional in a regulated environment — it is what model validators and auditors review first.

In Part 3, we begin the actual distributional analysis — examining each feature one variable at a time. You will see how the missing value decisions from this part directly shape what you can and cannot conclude from the distributions.
