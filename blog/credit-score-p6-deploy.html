Building and Monitoring a Credit Score Model

Deploy Model

Le Thanh Binh  ·  2026

The model has passed governance review and the business case has been signed off. Now comes the engineering work of making the credit score reliably available to every loan application processed by the bank's origination system. Deployment in banking is not a one-time event — it is a carefully staged rollout with defined checkpoints, fallback mechanisms, and a clear definition of what "done" means.

Packaging

Packaging and Serving the Model

Model serialization

  • Serialize the scikit-learn pipeline (preprocessing + model) using joblib or pickle; use ONNX format for cross-environment portability.
  • Store the serialized artifact in the model registry with a version tag, never on a local filesystem where it can be silently overwritten.
  • Load the artifact in a clean environment (no shared state with training) and verify identical predictions on a fixed test set — this catches library version mismatches.
  • A model serialized with scikit-learn 1.2 may behave differently under 1.4 if preprocessing behavior changed; test across dependency versions.

Serving architecture

  • REST API. Expose a scoring endpoint that accepts raw applicant features (application form + bureau pull) and returns probability of default, credit score band, and recommended decision (approve/decline/refer).
  • Docker containerization. Package the service with all dependencies pinned; deploy on Kubernetes (or equivalent orchestration) for horizontal scaling during peak periods.
  • SLA target. Response time must be within the loan origination system SLA — typically 500ms to 2 seconds including the upstream bureau call.
  • Peak load. Vietnamese retail banking sees 3–5x spike in application volume during month-end salary periods; sizing must account for this seasonality.

Feature store integration

Online Features
  • Real-time lookup of applicant bureau score (fresh pull)
  • Application form fields submitted by user
  • Device fingerprint / risk signals
Latency: < 50ms
Offline Features
  • Pre-computed aggregates (avg balance last 3m)
  • Missed payments in last 6 months
  • Transaction count in last 30 days
Refresh: nightly batch

Features requiring historical aggregation (e.g., average balance over 3 months, missed payment count) are read from the feature store rather than recomputed on the fly. The feature store pre-computes and caches values on a defined refresh schedule. The serving code must handle stale or missing values with a defined fallback: either use a conservative default, or escalate for manual underwriting.

Load Testing and Latency Budgets

  • Load test the scoring service at 2× expected peak traffic; define SLAs: P95 < 500ms, P99 < 1 second.
  • If targets are not met, identify the bottleneck: model inference (unlikely), feature store lookup, or bureau API call. Each has a different fix.
  • Instrument monitoring for all latency buckets; alert when any percentile approaches the SLA bound.
  • Run failure mode testing: what happens when the feature store is down, slow, or returns partial data?
Rollout Strategy

Staged Rollout

Shadow mode

  • Run the new model in parallel with the existing system for 2–4 weeks; process every application with both but log the new model's decisions without acting on them.
  • Answer critical questions: Are scores within expected distribution? Do any applicants fail to get a score (feature unavailability, edge cases)? How often does the new model agree with the old?
  • Review shadow mode data daily for the first week, then weekly; investigate anomalies (spikes in null scores, unexpected distribution shifts, product-specific disagreements) before advancing.
  • Advancement criteria: score distribution matches validation, null rate < 0.1%, disagreement rate explainable by model differences alone.

Canary release

Fig 3 — Canary release: traffic allocates 0% live (shadow) → 5% → 25% → 50% → 100% full cutover over 8 weeks.
  • Route a small fraction of live traffic (5–10%) to the new model; remaining traffic uses the existing system.
  • Monitor approval rates and score distributions for the canary segment against the control segment for 2–4 weeks; expand if metrics are within expected bounds and service is stable under load.
  • Progressive expansion: 10% → 25% → 50% → 100%; each stage lasts 1–2 weeks depending on data volumes and metric stability.

Full cutover and rollback

  • Switch 100% of traffic to the new model; keep the old model warm (not decommissioned) for a defined rollback window — typically 2 weeks.
  • If a critical issue emerges (e.g., systematic bias for a specific applicant segment not caught during audit), re-route traffic to the old model immediately without requiring a new deployment.
  • Pre-define rollback triggers: specific metric thresholds, incident severity levels, and approval thresholds. Make the rollback decision in advance, not under pressure after an incident.

In banking, the rollback decision should be pre-agreed: define, before the cutover, the specific conditions (metric thresholds, incident severity levels) that trigger an automatic rollback vs. a manual investigation. A rollback decision made under pressure after an incident is rarely made well. Making it in advance removes the uncertainty.

Reliability

Circuit Breaker and Fallback Modes

The scoring service must gracefully degrade when external dependencies fail:

Closed (Normal)
All dependencies healthy; requests flow normally. Bureau API responding, feature store online.
Open (Bureau Down)
Bureau API unavailable; fallback to application-data-only score. Lower accuracy but operational.
Half-Open (Recovery)
Testing if bureau is back; periodically retry with limited traffic until confirmed healthy.
  • Define circuit breaker thresholds: if bureau API fails 5 times in 10 requests, trip the circuit and switch to fallback score.
  • The fallback behavior (application-data-only score) must be tested under simulated outage conditions and approved by the risk team, since it may have a different accuracy profile.
  • Document the fallback accuracy impact: "Fallback score has 8% lower AUC but 99.99% availability vs. full model at 97% AUC and 99.5% availability."
