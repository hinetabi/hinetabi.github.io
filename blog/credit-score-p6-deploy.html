Building and Monitoring a Credit Score Model

Deploy Model

Le Thanh Binh  ·  2026

The model has passed governance review and the business case has been signed off. Now comes the engineering work of making the credit score reliably available to every loan application processed by the bank's origination system. Deployment in banking is not a one-time event — it is a carefully staged rollout with defined checkpoints, fallback mechanisms, and a clear definition of what "done" means.

Packaging

Packaging and Serving the Model

Model serialization

The scikit-learn pipeline (including preprocessing and the model itself) is serialized using joblib or pickle. For models that need to run across different environments or languages, ONNX format provides a more portable option. The serialized artifact is stored in the model registry with its version tag, not on a local filesystem where it can be silently overwritten.

One critical check: load the serialized artifact in a clean environment (no shared state with the training environment) and verify that it produces identical predictions on a fixed set of test cases. This catches library version mismatches — a model serialized with scikit-learn 1.2 may behave differently when deserialized under scikit-learn 1.4 if any preprocessing behavior changed between versions.

Serving architecture

For a credit scoring use case with real-time loan origination, the scoring service typically exposes a REST API. The request payload contains the applicant's raw features (from the application form and the bureau pull); the service preprocesses them through the pipeline and returns a probability of default, a credit score band, and a recommended decision (approve/decline/refer for manual review). Response time must be within the SLA of the loan origination system — typically 500ms to 2 seconds including the upstream bureau call.

The service is containerized in Docker with all dependencies pinned. It is deployed on Kubernetes (or an equivalent orchestration layer) for horizontal scaling during peak periods — month-end salary periods typically see a 3–5x spike in loan application volume in Vietnamese retail banking.

Feature store integration

For features that require historical aggregation (e.g., average balance over the past 3 months, number of missed payments in the past 6 months), the serving layer reads from a feature store rather than recomputing on the fly. The feature store pre-computes and caches these values with a defined refresh schedule. The serving code must handle the case where a feature store lookup returns a stale or missing value — with a defined fallback: either use a conservative default value, or escalate the application for manual underwriting.

Rollout Strategy

Staged Rollout

Shadow mode

The new model runs in parallel with the existing decisioning system for the first 2–4 weeks. Every application is processed by both the old system and the new model. The new model's decisions are logged but not acted upon. This period answers: does the new model produce scores within the expected distribution? Are there any applicants for whom the model fails to return a score (due to feature unavailability or edge case handling issues)? How often does the new model agree with the old system, and where does it disagree?

Shadow mode data is reviewed daily for the first week, then weekly. Anomalies — sudden spikes in null scores, unexpected score distribution shifts, high disagreement rates for a specific product type — should be investigated and resolved before advancing to the next phase.

Fig 1 — Staged rollout: shadow (observe only) → canary (partial traffic) → full cutover with a rollback path kept warm.

Canary release

Route a small fraction of live traffic (typically 5–10%) to the new model's decisions. The remaining traffic continues to use the existing system. Monitor approval rates and score distributions for the canary segment against the control segment for 2–4 weeks. If key metrics (approval rate, score distribution shape) are within expected bounds and the service is stable under load, expand the canary percentage progressively: 10% → 25% → 50% → 100%.

Full cutover and rollback

The full cutover switches 100% of traffic to the new model. At this point, the old model is kept warm (not decommissioned) for a defined rollback window — typically 2 weeks. If a critical issue emerges post-cutover (e.g., the new model is found to have a systematic bias for a specific applicant segment that was not caught during audit), traffic can be immediately re-routed to the old model without requiring a new deployment.

In banking, the rollback decision should be pre-agreed: define, before the cutover, the specific conditions (metric thresholds, incident severity levels) that trigger an automatic rollback vs. a manual investigation. A rollback decision made under pressure after an incident is rarely made well. Making it in advance removes the uncertainty.

Operations

Load Testing and Latency Budgets

Before the canary release, load test the scoring service at 2× the expected peak traffic. Define latency SLAs: P95 response time under 500ms, P99 under 1 second. If the service cannot meet these targets under load, investigate whether the bottleneck is the model inference itself (unlikely for a gradient boosted model — inference is fast), the feature store lookup latency, or the downstream bureau API call. Each bottleneck has a different remediation: model optimization, feature store caching, or circuit breaker patterns for the bureau API.

Define and instrument a circuit breaker: if the bureau API is unavailable, the scoring service should fall back to an application-data-only score (lower accuracy but operational) rather than failing the entire application. This fallback behavior must be documented, tested under simulated bureau outage conditions, and approved by the risk team — since the fallback score may have a different accuracy profile than the full model.
