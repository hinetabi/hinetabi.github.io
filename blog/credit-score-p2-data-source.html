Building and Monitoring a Credit Score Model

Data Source Check

Le Thanh Binh  ·  2026

With the problem defined in Part 1, the next question is: what data does the bank actually have, and how much of it is usable? In practice, a significant portion of data engineering effort in credit scoring is spent answering this question — not because the data does not exist, but because it is spread across multiple systems with inconsistent formats, ownership boundaries, and freshness guarantees.

Source Inventory

The Four Data Source Categories in Banking

Most retail banks have access to four broad categories of data that are relevant to credit scoring. Each carries a different type of signal and a different set of operational constraints.

1. Application data

Collected at the time the borrower submits a loan application: declared income, employment status, tenure, requested loan amount, loan purpose, existing liabilities. This data is self-reported, which means it may be inaccurate or subject to fraud. It is, however, always available at inference time because the borrower fills it in. Application data typically forms the backbone of a scorecard, especially for new-to-bank customers where behavioral history is limited.

2. Credit bureau data

External data pulled from a credit bureau (e.g., CIC in Vietnam, CIBIL in India, Experian globally) at the time of application. It includes: existing credit obligations, repayment history across lenders, number of open accounts, recent credit inquiries, and in some markets a pre-computed bureau score. Bureau data is powerful — it captures behavior across the borrower's entire credit history, not just within the bank. The key constraints are cost (each bureau pull has a fee), coverage (thin-file or new-to-credit borrowers may have minimal bureau records), and regulatory compliance around what bureau attributes can be used in an automated credit decision.

3. Internal transaction and behavioral data

For existing bank customers (as opposed to new applicants), the bank has visibility into account activity: salary credits, spending patterns, average daily balance, ATM withdrawal frequency, bill payment behavior, and product holdings (savings account, existing loans, insurance). This is proprietary data that no other lender has access to, which makes it a genuine competitive moat. A customer who maintains a consistently high average balance and receives regular salary credits is a materially different risk profile from one who does not — even if their bureau score is identical.

4. Alternative data

In markets with large thin-file populations (young adults, self-employed individuals), alternative data can fill the gap where bureau records are sparse. This includes: telco data (call patterns, top-up frequency, SIM tenure), e-commerce transaction history, mobile app usage signals, and utility payment records. Alternative data requires careful legal and ethical review — data sharing agreements, consent frameworks, and in many jurisdictions explicit regulatory approval. The signal quality also varies significantly and requires empirical validation rather than assumption.

Source Audit

Auditing Each Source Before Feature Engineering

For each candidate data source, a structured audit answers four questions before any feature is built:

Fig 1 — Data source register: coverage, signal strength, serving latency, and compliance status

Coverage

What fraction of the target population (loan applicants) has a record in this source? Bureau coverage is often below 100% — particularly for first-time borrowers or customers from rural areas. Internal transaction data only exists for customers who already hold a bank account. If a source covers only 40% of applicants, any feature derived from it will require a missingness strategy for the remaining 60%, which can introduce bias if the uncovered segment is systematically different from the covered one.

Relevance

Does this source plausibly carry predictive signal for default behavior? A quick univariate analysis (comparing the distribution of a feature between good and bad borrowers) is sufficient at this stage. Information Value (IV) is a standard credit risk metric for this: IV below 0.02 is typically considered weak, 0.02–0.1 medium, above 0.1 strong. Use IV as a fast filter, not a final selection criterion — features with moderate IV can still contribute in combination.

Access and compliance

Who owns this data source? Is there a data sharing agreement in place? Does using this data in an automated credit decision comply with local consumer credit regulations? In regulated markets, certain attributes (gender, ethnicity, religion, marital status, pregnancy) are explicitly prohibited as model inputs. Beyond explicit prohibitions, regulators increasingly scrutinize proxy variables — features that are nominally neutral but correlate strongly with a protected class. Document the legal basis for using each source before feature engineering begins.

Latency and freshness

Can this data be served at the time a credit decision is needed? Bureau pulls happen in near-real-time in most markets. Internal transaction aggregates may require a T+1 batch job. If a feature requires a batch computation that runs at midnight, it cannot support an application submitted at 11 PM unless the feature store caches yesterday's value — which introduces a staleness assumption. For each feature, define: what is the maximum acceptable staleness, and what happens if the data source is unavailable (fallback value, feature imputation, or decline the application with an error code)?

A feature that exists in training data but cannot be reliably served in production will become a silent liability. The training-serving consistency check is not optional. The same transformation pipeline — including imputation logic, encoding, and scaling — must be executable both offline (training) and online (inference). Discrepancies here are one of the most common causes of model degradation that is hard to diagnose post-deployment.

Output

Data Source Register

The output of this stage is a data source register: a table documenting every candidate source with its coverage rate, estimated IV, owner, compliance status, latency, and a go/no-go decision for inclusion in the feature engineering phase. Sources that are excluded should have a documented reason — this prevents the same discussion from happening again in six months when a new team member joins the project.

For a typical retail bank in a Southeast Asian market, the register might include: application form fields (go, always available), CIC bureau score and attributes (go, subject to coverage), internal salary and balance aggregates (go, existing customers only), and telco data (conditional, subject to data sharing agreement and regulatory review). This clarity at the source level makes the EDA phase in Part 3 much more focused and efficient.
