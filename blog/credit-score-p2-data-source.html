Building and Monitoring a Credit Score Model

Data Source Check

Le Thanh Binh  ·  2026

With the problem defined in Part 1, the next question is: what data does the bank actually have, and how much of it is usable? In practice, a significant portion of data engineering effort in credit scoring is spent answering this question — not because the data does not exist, but because it is spread across multiple systems with inconsistent formats, ownership boundaries, and freshness guarantees.

Source Audit

Auditing Each Source Before Feature Engineering

For each candidate data source, a structured audit answers four questions before any feature is built:

Q1Coverage
  • What % of loan applicants have a record here?
  • Is the uncovered segment systematically different?
  • Bureau: thin-file & rural applicants often missing
  • Internal data: new-to-bank applicants have no history
  • Sub-50% coverage → requires explicit missingness strategy
Q2Signal Relevance
  • Does this source carry predictive signal for default?
  • Quick check: Information Value (IV) univariate analysis
  • IV < 0.02 → weak, skip for initial feature set
  • IV 0.02–0.1 → medium, include for evaluation
  • IV > 0.1 → strong, prioritize for feature engineering
Q3Access & Compliance
  • Who owns the data? Is a sharing agreement in place?
  • Prohibited attributes: gender, ethnicity, religion, pregnancy
  • Beware proxy variables that correlate with protected classes
  • Document the legal basis before feature engineering
  • In Vietnam: SBV Model Risk Management circular applies
Q4Latency & Freshness
  • Can this data be served when the decision is made?
  • Bureau: near-real-time in most markets
  • Batch features: need feature store cache (T+0)
  • Define: max acceptable staleness per feature
  • Define: fallback if source is unavailable (impute / refer)

Critical Check

Training-Serving Consistency

The most insidious failure mode in credit scoring deployment is a training-serving gap: a feature that exists in training data but is not reliably available — or is computed differently — at serving time. The same transformation pipeline must be executable both offline and online.

CORRECT: Same pipeline, both environments

Offline Training
Raw features → impute → WOE encode → scale → model.fit()
≡
Online Serving
Raw features → impute → WOE encode → scale → model.predict()

DANGEROUS: Diverged pipelines

Offline Training
avg_balance_6m computed over full history in SQL
≠
Online Serving
avg_balance_6m fetched from feature store — stale by 24h
Fig 2 — Training-serving consistency. Diverged pipelines produce silent model degradation that is hard to diagnose post-deployment.

Common causes of training-serving gaps to check for every feature:

  • Transformation mismatch — imputation applied in training but not in the serving pipeline, or vice versa
  • Staleness assumptions — batch-computed features cached from yesterday do not match the real-time context of today's application
  • Aggregation window differences — "last 6 months" means different things depending on whether the anchor date is the application date or the current date
  • Encoding drift — WOE bins fitted on training data may not handle new categories seen in production (e.g., a new employment type added to the application form)

Every feature that exists in the training set must have a documented serving path. If a serving path cannot be established before feature engineering begins, exclude the feature — not after training, when removing it costs days of rework.

Output

Data Source Register

The output of this stage is a register documenting every candidate source. For each source include:

  • Coverage rate across the target application population
  • Estimated IV from a quick univariate analysis
  • Owner & access path (which team, which API or table)
  • Compliance status (cleared / under review / excluded)
  • Serving latency and the fallback behavior if unavailable
  • Go / No-go decision with reason if excluded

Sources excluded should have a documented reason — this prevents the same discussion happening again in six months when a new team member joins the project.
